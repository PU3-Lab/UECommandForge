# Harness 범위와 우선순위 기준 (Harness Scope & Priorities)

본 문서는 UECommandForge harness가 **무엇을 자동화하고, 무엇을 자동화하지 않는가**, 그리고 **다음에 무엇을 만들지**를 판정하는 기준을 정의한다. 모든 신규 커맨드렛은 이 문서의 경계와 루브릭을 통과해야 한다.

> 관련 문서: [cpp_vs_blueprint_decision.md](cpp_vs_blueprint_decision.md) — C++ vs Blueprint 생성 판단. 본 문서는 그 상위의 "자동화 대상 선정" 기준이다.

---

## 1. 미션

> **사람이 자주 반복하는 작업과 실수하기 쉬운 부분을 최소화한다.**

harness는 커맨드렛 기반 UE 플러그인으로, 기계적·규칙적·반복적 작업을 결정적으로 수행하고, 되돌릴 수 없는 실수를 적용 전에 차단한다. **창작과 판단은 사람이 유지한다.**

---

## 2. 핵심 원리

- AI는 UE를 직접 보지 않는다. **커맨드렛이 Result JSON으로 직렬화해 준 것만** 인식·검토한다. → 리뷰 가능 범위 = Result JSON 스키마.
- harness의 **쓰기 경계 = 자동화 목표와 같은 선**이다. 못 하는 영역(창작·로직·판단)은 애초에 자동화 대상이 아니다.
- harness가 수행하는 동사는 셋뿐: **Create(껍데기/데이터) · Configure(프로퍼티/import 설정) · Validate**. **Author(창작)와 Wire(로직)는 범위 밖.**

---

## 3. 에셋 쓰기 능력 매트릭스

| 능력 | 영역 | 예 |
|---|---|---|
| ✅ **가능** | 리플렉션 프로퍼티 set + Factory 생성 + import | Blueprint 껍데기(클래스/컴포넌트/CDO 기본값), DataTable/DataAsset/StateTree, Material Instance 파라미터, 레벨 액터 배치, 폴더, import 설정, Config 값 |
| ⚠️ **부분 (brittle·제한적)** | API는 있으나 복잡 구조는 고통 | Material 그래프 노드, Landscape 하이트맵 import(스컬프팅 ❌), Sequencer(키프레임 저작 제한), Blueprint 변수/함수 **시그니처**(본문 ❌) |
| ❌ **불가** | 그래프 로직·창작·베이크·인터랙티브 | 아래 4영역 |

### ❌ 불가 4영역

1. **그래프 로직 / 노드 네트워크** — Blueprint Event Graph, AnimGraph, Niagara, Control Rig, PCG, Metasound. 노드 스폰·핀 연결 API 부재/사실상 막힘.
2. **창작 콘텐츠 (import만 가능)** — 메시 지오메트리, 리그, 스킨 웨이트, 애니메이션 키, 텍스처 픽셀. DCC에서 만들어 들여올 뿐 harness가 창작 불가.
3. **빌드/베이크 산출물 (트리거만 가능)** — 라이트맵, Navmesh, HLOD/프록시, Nanite/VT 빌드 데이터, 쿡 콘텐츠/DDC/셰이더 바이트코드.
4. **인터랙티브 에디터 전용 (헤드리스 API 없음)** — Landscape 스컬프팅, Foliage 페인팅, Mesh Paint, Modeling Mode.

> 추가: 기존 바이너리 `.uasset`은 **리플렉션 프로퍼티 값 변경**까지만. 구조 변경(그래프 재배선, 노드망 교체)·바이너리 머지 충돌 해결은 ❌.

## 4. "반복 × 실수" 우선순위 루브릭

다음에 만들 커맨드렛은 **커버리지가 아니라 빈도 × 실수율 × 자동화 가능성**으로 고른다.

| 축 | 질문 | 점수(0–3) |
|---|---|---|
| **빈도(Frequency)** | 사람이 이 작업을 얼마나 자주 반복하나? | 0=드묾 … 3=매일 |
| **실수율(Error rate)** | 손으로 하면 얼마나 자주·치명적으로 틀리나? | 0=무해 … 3=빌드/출시 차단 |
| **자동화 가능성(Automatable)** | 직렬화·규칙화 가능하고 §3에서 ✅/⚠️인가? | 0=불가(❌) … 3=완전 ✅ |

- **우선순위 = 빈도 + 실수율 + 자동화 가능성** (자동화 가능성이 0이면 채택 불가).
- 높은 빈도 + 높은 실수율 + 직렬화 가능 → **최우선**.
- 창작/판단 영역(§5)이거나 자동화 가능성 0 → **자동화 안 함**.

### 적용 예시

| 후보 | 빈도 | 실수율 | 자동화 | 합 | 판정 |
|---|---:|---:|---:|---:|---|
| 네이밍·경로 정책 검증 | 3 | 3 | 3 | 9 | 채택됨 (구현 완료) |
| 참조 연결(Character↔AIController↔StateTree) | 2 | 3 | 3 | 8 | 채택됨 (구현 완료) |
| 플랫폼 Config Diff / Packaging 검증 (Phase 9) | 2 | 3 | 3 | 8 | **다음 타깃** |
| Import 설정 일관성(텍스처 압축/LOD) | 3 | 2 | 3 | 8 | 후보 |
| AnimGraph 로직 생성 | 2 | 2 | 0 | — | 제외 (자동화 불가) |
| UMG 위젯 바인딩 생성 | 2 | 2 | 0 | — | 제외 (자동화 불가) |

## 5. 사람 게이트 (Human-in-loop 고정)

다음은 harness가 판단하지 않고 **항상 사람이 결정**한다.

- 아트 디렉션, 게임 feel/재미, 레벨 디자인, 오디오 믹스, UX 직관성.
- 시각적 최종 품질(스크린샷 1차 판독은 보조일 뿐).
- 타깃 하드웨어 성능 판정, 콘솔 플랫폼 인증.
- 되돌릴 수 없는 에셋 변경의 **적용 승인** (preflight → 승인 → 적용 → rollback 계약).

---

## 6. 신규 커맨드렛 채택 체크리스트

새 커맨드렛을 제안할 때 다음을 모두 충족해야 한다.

- [ ] §3에서 ✅ 또는 ⚠️ 영역인가? (❌면 중단)
- [ ] §4 루브릭 점수가 충분한가? (자동화 가능성 ≥ 1 필수)
- [ ] 입력 Spec / 출력 Result JSON / 실패 조건이 명확한가?
- [ ] 파괴적 변경이면 preflight·승인·rollback 경로가 있는가?
- [ ] §5 사람 게이트 영역을 침범하지 않는가?
