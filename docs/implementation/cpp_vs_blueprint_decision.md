# C++ vs Blueprint 의사결정 기준표 (C++ vs Blueprint Decision Guide)

본 문서는 Codex 및 LLM 에이전트 자동화 파이프라인에서 작업 시간을 최소화하기 위해 "언제 C++ 클래스를 생성하고, 언제 Blueprint로 바로 대체할 것인가"를 판단하는 의사결정 기준을 정의한다.

---

## 1. 개요 및 핵심 원칙
C++ 클래스를 새로 작성하고 반영하는 과정은 **UBT 컴파일 오버헤드(최소 10~15초)**와 **바이너리 빌드 상태 리스크**를 수반한다. 반면, Blueprint 에셋 생성은 오버헤드가 없는 즉각적인 에셋 패키징이 가능하다.

*   **기본 규칙 (Default-Blueprint):** 특별한 이유가 없다면 기본적으로 **Blueprint-Only** 생성을 지향한다.
*   **예외 규칙 (Exceptional-C++):** 아래의 **C++ 필수 조건** 중 하나 이상을 충족하는 경우에만 C++ 클래스 생성을 트리거한다.

---

## 2. 의사결정 기준표 (Decision Matrix)

| 구분 | C++ 클래스 생성 필수 (Blueprint 대체 불가) | Blueprint-Only 대체 권장 (C++ 불필요) |
| :--- | :--- | :--- |
| **상속 계층** | - 신규 네이티브 인터페이스(`UInterface`) 선언이 필요할 때<br>- 다중 C++ 상속이나 구조적 다형성이 요구될 때 | - 엔진 기본 클래스(`Actor`, `Character`, `Pawn`, `AIController` 등) 상속만으로 충분할 때 |
| **리플렉션 및 통신** | - 신규 `UFUNCTION(BlueprintCallable)` 또는 `UPROPERTY(EditAnywhere)` 등 완전히 새로운 네이티브 속성/함수 노출이 요구될 때 | - 기존 C++ 부모 클래스가 이미 제공하는 속성과 함수를 재사용하여 오버라이드할 때 |
| **의존성** | - 타 플러그인 또는 C++ 모듈(`Build.cs` 의존성 추가)을 참조해야 할 때 | - 기존 프로젝트/플러그인 설정 및 모듈 범주 내에서 조립할 수 있을 때 |
| **동작 및 성능** | - 물리 연산, 실시간 틱(Tick) 처리, 대규모 수학 공식 연산 등 성능이 매우 중요한 시스템을 정의할 때 | - 컴포넌트(메쉬, 오디오, 콜리전 등)를 부착하고, 디폴트 파라미터 값을 수정하는 조립 위주의 구성일 때 |
| **수명 주기** | - 프로젝트 전반에 걸쳐 공통 프레임워크 기반 구조로 장기 유지보수되어야 할 네이티브 뼈대를 정의할 때 | - 빠른 프로토타이핑, 맵 배치, AI Flow(StateTree 연결) 조합, 단순 기능 테스트용 액터 생성 시 |

---

## 3. 에이전트(Codex) 판단 알고리즘
Codex 에이전트는 사용자의 작업을 지시받았을 때 다음과 같은 의사결정 트리를 따른다.

```text
사용자 요구사항 분석
   ↓
[Q1] 새로운 네이티브 멤버 변수/함수(UPROPERTY, UFUNCTION) 선언이 필요한가?
   ├─ YES → [C++ 생성]
   └─ NO  → [Q2] 엔진 기본 제공 클래스나 기존 C++ 부모 클래스로 대체 가능한가?
               ├─ YES → [Blueprint-Only 생성] (즉시 적용, UBT 스킵)
               └─ NO  → [C++ 생성]
```

이 판단 로직을 적용함으로써 에이전트의 불필요한 C++ 빌드 유발을 사전 차단하고, 전체 자동화 실행 효율을 극대화한다.
