## 2. 발견 사항 (Findings)

### ✅ F1(R1 확인) — `command` 키

- `JsonReportWriter.cpp:16`에서 `commandlet`과 함께 `command`를 직렬화. 테스트도 `report has command key` 단언(test `:178`). plan 스모크의 `.command` jq가 이제 유효.

### ✅ F2(R2 확인) — 엔진 Editor 전용 플러그인 오탐 해소 + 회귀 테스트

- `ResolvePluginDescriptor`가 프로젝트 Plugins → 엔진 Plugins 순으로 `.uplugin` 해석(`cpp:104-123`). forbidden 분기가 `DescriptorHasRuntimeModule`로 editor-only를 제외(`cpp:340-345`), 미해석 시 사유 명시 후 보수적 위반(`cpp:347-350`).
- **회귀 테스트 신규**: `FValidatePluginDepsEngineEditorOnlyN
otForbiddenTest`(test `:302`)가 `-TestEnginePluginsDir=`로 가짜 엔진 디렉터리에 editor-only `.uplugin`을 두고 forbidden+Shipping에서 미발생을 단언. _02 R2가 지적한 "테스트가 엔진 경로를 못 잡음"이 해소됐다. 테스트 주입용 `TestEnginePluginsDir` 파라미터(`cpp:221`) 설계도 적절.

### ✅ F3(R3 확인) — IPluginManager 제거, 대상 프로젝트 기준

- `IPluginManager::Get().GetDiscoveredPlugins()` 완전 제거. `LoadEnabledPlugins`는 `.uproject` refs(`FPluginReferenceDescriptor::IsEnabledFor*`) + **정책에 명시된 이름만** lazy descriptor 해석(`cpp:185-203`)으로 enabled-by-default 평가 — 전수 엔진 스캔 회피, 효율적.
- editor-module warning 루프가 **대상 프로젝트 `<ProjectDir>/Plugins/**`** 의 `.uplugin`을 순회(`cpp:360-364`)하도록 변경 — 실행 에디터 마운트 목록 누수 제거.

### ✅ F4(D1 확인) — 외부 프로젝트 config 조준 실증

- `FDiffPlatformConfigExternalProjectTest`(test `:259-309`)가 `TempDir()/ExtProj/`에 `T.uproject` + 외부 전용 키(`ExternalOnlyKey`, Windows/Mac 상이)를 합성하고 `-Project=ExtProjPath`로 실행해 diff 감지를 단언.
- 양성/음성 논리가 성립: `ExternalOnlyKey`는 실행(샘플) 프로젝트에 없으므로, 만약 실행 프로젝트를 읽었다면 diff 0 → `ok:true` → 테스트 FAIL. 통과 시 외부 config가 실제 로드됨이 증명된다. _04 R1(D1) 해소.

### ❌ F5(R5 미해결) — 작업 트리에 Engine.ini 오염 잔존 (읽기 전용 원칙 의심)

- 수정 커밋 이후에도 `git status`에 `sample/Config/DefaultEngine.ini`, `Mac/MacEngine.ini`, `Windows/WindowsEngine.ini`와 다수 `.uasset` 수정이 **미커밋**으로 남아 있다.
- `git diff` 결과 Engine.ini 변경은 **빈 줄만 추가**(의미 없는 trailing newline). 테스트 fixture는 `*UECommandForgeTest.ini`만 쓰고 cleanup하므로, 이 Engine.ini 변경은 **config 로드 경로의 디스크 writeback 부작용**일 가능성이 있다.
- 우려: DiffPlatformConfig는 `bDryRun=true` 읽기 전용을 표방하는데, 실행 시 소스 `Engine.ini`가 디스크에서 수정된다면 **읽기 전용 원칙 위반**이다(`_01`/`_03`의 "무파괴 원칙 엄격 준수" 주장과 모순 가능).
- 권장:
  1. 이 Engine.ini 변경이 어디서 발생하는지 추적(테스트/커맨드렛 config 로드 후 flush 여부). `FConfigContext`/`FConfigFile`이 dirty 상태로 디스크에 반영되지 않도록 보장.
  2. 현재 오염분은 `git checkout -- sample/Config sample/Content`로 복원 후, 재발하지 않음을 확인.

### 🟢 F6(Minor) — editor-module warning 사각지대

- warning 루프는 `EnabledPlugins.Find(PluginName)`로 게이트(`cpp:377-381`)한다. 그러나 `EnabledPlugins`는 `.uproject` refs + **정책 명시 플러그인**만 채워진다(`cpp:185`). 따라서 `.uproject`에도 정책에도 없지만 **EnabledByDefault로 켜진 프로젝트 플러그인**(예: in-tree 플러그인)은 map에 없어 warning 평가에서 제외된다.
- 영향: 런타임+에디터 모듈 혼재 경고의 커버리지 공백. 권장: warning 루프는 자체적으로 `.uplugin`의 EnabledByDefault/플랫폼 조건을 평가하거나, 프로젝트 Plugins 전수를 enabled 집합에 반영.

### 🟢 F7(Minor) — `EnabledByDefault == Enabled` 단정

- enabled-by-default 판정이 `Desc.EnabledByDefault == EPluginEnabledByDefault::Enabled`(`cpp:196`)만 인정한다. `EPluginEnabledByDefault::Unspecified`(엔진 플러그인 다수)는 컨텍스트상 활성일 수 있는데 비활성으로 처리되어 required 판정에서 **false negative** 여지가 있다. 정책에 엔진 플러그인을 required로 넣는 경우 확인 필요.
