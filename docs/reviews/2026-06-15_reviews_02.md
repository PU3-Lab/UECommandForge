## 2. 재리뷰 발견 사항 (Re-findings)

### 🔴 R1 — H1 "command 키 추가" 주장은 사실과 다름

- _01 §3 H1은 "리포트 출력에 `commandlet`과 함께 `command` 키를 추가로 직렬화"했고 `jq -e '.command == "ValidatePluginDependencies"'`가 통과한다고 기술한다.
- 그러나 직렬화는 여전히 `commandlet` **단일 키**다 — `JsonReportWriter.cpp:15` (`Root->SetStringField(TEXT("commandlet"), ...)`). `command` 키는 코드에 존재하지 않는다.
- 실제 스모크(`tools/test/smoke/validate_plugin_deps.sh:40,69-70`)는 `.command`를 **검사하지 않고** `.ok` / `.issues[].code`만 본다. 즉 스모크가 그린인 이유는 "필드 추가"가 아니라 "검사 항목을 바꿔 회피"한 것이다.
- 영향: 계획서(plan §Task7 Step5)의 원래 `.command` jq는 **여전히 실패**한다. _01의 M1 "snake_case 통일(`command` 포함)" 주장도 같은 이유로 부정확하다.
- 권장: `command` 키를 실제 추가하거나, 스펙 §2.3과 plan 스모크 문구를 `commandlet` 기준으로 정정해 문서·코드·스모크를 일치시킨다.

### 🔴 R2 — H2 오탐은 forbidden 경로에서 구조적으로 미해결

- forbidden-in-shipping 판정이 호출하는 `PluginHasRuntimeModule`은 **변경되지 않았다** — 여전히 대상 프로젝트 `Plugins/`만 검색하고(`cpp:181-185`), `.uplugin`을 못 찾으면 보수적으로 `true` 반환(`cpp:185`).
- forbidden 분기(`cpp:328-329`)는 이 헬퍼를 그대로 사용한다. 따라서 엔진 Editor 전용 플러그인(`EditorScriptingUtilities` 등)이 `forbiddenInShipping`에 있고 활성으로 계산되면 → 프로젝트 디렉터리에서 미발견 → `true` → **여전히 `PLUGIN_FORBIDDEN_IN_SHIPPING_ENABLED` 오탐**.
- `IPluginManager` 연동은 *활성 계산*과 *별도의 Editor 모듈 warning 루프*(`cpp:340-396`)에만 적용됐고, **오탐의 실제 원인인 forbidden 헬퍼에는 닿지 않았다.**
- 회귀 테스트도 여전히 *프로젝트 내* 합성 플러그인(`MyEditorTool`)만 커버한다. 스모크 case 1(`validate_plugin_deps.sh:21-40`)이 `EditorScriptingUtilities`를 forbidden+Shipping으로 돌려 `.ok == true`를 기대하는데, 이 그린은 "Editor 전용 판정이 동작해서"가 아니라 **해당 엔진 플러그인이 비활성으로 계산될 때만** 성립한다(매우 취약한 전제). 엔진 플러그인 descriptor의 `EnabledByDefault` 해석에 따라 결과가 갈리며, 본 세션에서 실기 확인 불가.
- 권장: forbidden 판정의 모듈 타입 조회를 `IPluginManager`의 descriptor 기반으로 통일하거나, `PluginHasRuntimeModule`에 엔진 플러그인 경로(`FPaths::EnginePluginsDir()`)를 포함. + 엔진 Editor 전용 플러그인 오탐 회귀 테스트 추가.

### 🔴 R3 — (신규) 외부 `-Project` 검증인데 "실행 중 프로젝트"를 스캔

- H2 수정이 도입한 `IPluginManager::Get().GetDiscoveredPlugins()`(`cpp:124`, `cpp:285`)는 **현재 실행 중인 커맨드렛(샘플 프로젝트)의 엔진+프로젝트 마운트 상태**를 반환한다. 검증 대상인 외부 `-Project`의 플러그인 집합이 아니다.
- 결과(외부 프로젝트 검증 시):
  - enabled-by-default 주입(`cpp:153-171`)이 대상이 아니라 **실행 중 프로젝트의 디스커버 결과**를 대상 enabled 집합에 합친다.
  - Editor 모듈 warning 루프(`cpp:340-353`)는 `GetLoadedFrom()==Project` 필터로 **실행 중 샘플 프로젝트의 플러그인**(예: UECommandForge 자체)을 스캔한다. 대상 프로젝트의 플러그인이 아니다.
  - 실행 중 프로젝트의 고유 플러그인이 대상의 enabled 집합으로 **누수**된다.
- 즉 `-Project == 실행 중 샘플`일 때만 우연히 맞고, 커맨드렛의 본래 목적(임의 외부 프로젝트 검증)에서는 잘못된 결과를 낸다. 기존 파일 검색 방식이 이 점에서는 오히려 정확했다.
- 권장: 외부 프로젝트 검증은 `IPluginManager`(실행 컨텍스트 의존)가 아니라, 대상 프로젝트/엔진 디렉터리의 `.uplugin`을 직접 해석하는 경로로 일원화한다. `IPluginManager`를 쓰려면 "대상 == 실행 프로젝트" 전제를 문서·인자 검증으로 강제해야 한다.

### ⚠️ R4 — DiffPlatformConfig: 스펙 H1 스파이크 증빙 부재

- 구현은 스펙 §4.3대로 `FConfigContext::ReadIntoLocalFile(File, Platform)`(`DiffPlatformConfigCommandlet.cpp:211`)을 사용한다 — 방향은 맞다.
- 그러나 스펙 §9-0이 **선행 필수**로 못 박은 "단일 호스트(Mac)에서 `FConfigContext`가 비호스트 플랫폼(Windows/Linux)의 Base/Default/<Platform> 계층을 정확히 해석하는가"에 대한 **스파이크 결과가 어디에도 기록되지 않았다.**
- 비호스트 플랫폼 effective value가 부분만 해석되면 diff 결과가 조용히 틀리며, 이는 스펙이 명시한 H1 리스크 그 자체다.
- 권장: 스파이크 증빙(샘플 플랫폼별 effective value 덤프 비교)을 리포트/문서로 남긴 뒤 PASS 판정. 미검증 상태로 PASS 불가.

### 🟡 R5 — "74 tests / 2 smoke 그린, 실기 검증 완료" 선언의 신뢰성

- R1(코드에 없는 `command` 키를 "추가했다"고 기술)·R2(미해결 오탐을 "원천 차단"으로 기술)에서 보듯, _01 문서의 조치 서술과 실제 코드가 어긋난다. 따라서 동일 문서의 "실기 검증 완료/PASS" 선언도 **그대로 신뢰하기 어렵다.**
- 본 재리뷰는 UE 실기 재현을 하지 않았으므로 "테스트가 실패한다"고 단정하지 않는다. 다만 R2/R3가 커버되지 않은 현재 테스트 구성으로는 그린이라도 **오탐/외부 프로젝트 결함을 잡지 못한다**(거짓 그린 가능).
