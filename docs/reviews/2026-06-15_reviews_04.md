## 2. 재리뷰 발견 사항 (Re-findings)

### ⚠️ R1(D1 잔여) — 외부 프로젝트 지원이 코드에만 있고 테스트로 증명되지 않음

- 조치는 실재한다: `Context.ProjectRootDir`/`Context.ProjectConfigDir`에 대상 경로를 주입(`cpp:218-224`).
- 그러나 4개 테스트 전부 `-Project`에 `FPaths::GetProjectFilePath()`(**실행 중 샘플 프로젝트**)를 넘기고(test `:104,142,194,234`), fixture는 `FPaths::ProjectConfigDir()`(실행 중 프로젝트 Config)에 ini를 쓴다(test `:37-39`).
- 즉 "대상 == 실행 프로젝트" 경로만 검증된다. 이 경우 ProjectConfigDir 주입값이 기본값과 같아, **주입 로직이 외부 프로젝트에서 올바른지(그리고 `ReadIntoLocalFile` 이후 멤버 override가 실제 로드 hierarchy에 반영되는지)는 여전히 미검증**이다.
- 잠재 리스크: `FConfigContext`가 `ReadIntoLocalFile` 시점에 경로/branch를 선계산한다면, 사후 `ProjectConfigDir` 대입이 부분만 반영될 수 있다(외부 프로젝트에서 조용히 실행 프로젝트 config로 폴백).
- 권장: **실제로 다른 디렉터리의 임시 프로젝트**(별도 `Config/` + `Platforms/`)를 `-Project`로 주는 회귀 테스트 추가. 그 프로젝트에만 존재하는 키가 diff에 잡히는지로 외부 조준을 단언.

### ✅ R2(D2 확인) — H1 스파이크가 테스트로 실증됨

- fixture가 디스크에 실제 플랫폼 ini를 생성한다: `Config/DefaultUECommandForgeTest.ini`(CommonValue), `Config/Windows/WindowsUECommandForgeTest.ini`(WindowsValue), `Config/Mac/MacUECommandForgeTest.ini`(MacValue) — test `:42-52`.
- 커맨드렛을 `-Platforms=Windows,Mac -Categories=UECommandForgeTest`로 실행해 값 차이를 단언(test `:111-122`). 이는 `FConfigContext`의 플랫폼별 hierarchy(Default+Platform override) 로드 경로를 실제로 태운다.
- _03 D2(인메모리 합성이라 미검증)의 핵심 우려가 해소됐다. _03 문서의 스파이크 증빙 narrative도 이 fixture와 일치한다.
- 잔여 주의: 본 재리뷰는 테스트를 직접 실행하지 않았으므로 "그린"은 코드 구조 기준 판단이다. 실기 로그로 1회 확정 권장.

### ✅ R3(D3 확인) — 배열 다중값 비교 구현됨 (단, 정렬 비교의 의미 1건)

- `Sec->MultiFind(*Key, RawValues)`로 다중값 수집 후 정렬·요소 비교(`cpp:294,304-327`). fixture가 `+Paths` 배열 차이를 주입(test `:74-79`).
- 설계 주의(결함 아님): `Array.Sort()` 후 비교라 **순서만 다른 차이는 의도적으로 무시**된다. 대부분의 UE 배열 config(집합 성격)에는 타당하나, _03 PASS 문서의 "추가·누락·값 불합 누수 없이 검출" 표현은 *순서 변경*은 제외됨을 함께 적어야 정확하다.

### ✅ R4(D4/D5/D6 확인)

- D4: `DetectsValueDiff`/`DetectsMissingKey`/`FiltersWithAllowlist`/`DetectsArrayKeyDiff` 4개로 분리(test `:92,130,182,222`), 전역 allowlist 편법 제거. 테스트 전용 카테고리(`UECommandForgeTest`)로 노이즈 격리.
- D5: allowlist 엔트리 `section`/`key` 결손 시 `CONFIG_ALLOWLIST_INVALID` 기록·중단(`cpp:76-84,107-115`).
- D6: 카테고리 로드 실패가 `ValidationFailed` 반환(`cpp:241-242`)으로 스펙 §4.5 정렬, 사문화 `HasErrorIssue` 헬퍼 제거 확인.

### 🟢 R5 — 작업 트리 위생 (신규)

- 변경분이 **미커밋**이며, `git status`에 `sample/Config/DefaultEngine.ini`, `sample/Config/Mac/MacEngine.ini`, `sample/Config/Windows/WindowsEngine.ini`와 다수 `.uasset`(BP_TestCharacter 등) 수정이 섞여 있다.
- 테스트 fixture는 `*UECommandForgeTest.ini`만 쓰고 cleanup하므로, 위 Engine.ini/uasset 수정은 **테스트 실행 부산물 또는 무관 변경**일 가능성이 높다. 커밋 시 샘플 실제 config가 오염될 수 있다.
- 권장: 커밋 전 `git diff`로 Engine.ini/uasset 수정이 의도된 것인지 선별. 의도치 않으면 `git checkout`으로 복원.
