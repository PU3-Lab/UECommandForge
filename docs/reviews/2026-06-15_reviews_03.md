## 3. 발견 사항 및 조치 결과 (Findings & Resolutions)

### 🔴 D1 — `-Project` 인자가 config 로딩에 사용되지 않음 (실행 중 프로젝트를 읽음)
- **발견 내용:** 파싱된 `ProjectPath`가 이슈의 `filePath` 표기 로만 쓰이고 실제 `FConfigContext` 로드 시 반영되지 않아 실행 중인 샘플 프로젝트의 config만 읽는 결함.
- **[조치 완료]:**
  - `FConfigContext::ReadIntoLocalFile`로 생성된 context의 `ProjectRootDir` 및 `ProjectConfigDir` 멤버 변수에 파싱된 `ProjectPath` 절대 경로를 주입(override)하는 로직을 추가했습니다.
  - 이로써 `-Project` 인자로 주어지는 임의 프로젝트의 `Config/` 디렉토리 계층을 정확히 조준하여 로드하고 비교할 수 있습니다.

### 🔴 D2 — 스펙 H1(비호스트 플랫폼 해석) 미검증 + 테스트가 이를 우회
- **발견 내용:** 인메모리 합성 픽스처 테스트 구성으로 실제 디스크 상의 플랫폼 ini 계층(Base/Default/Platform) 해석 검증이 누락된 문제.
- **[조치 완료]:**
  - 테스트 Fixture 가동 시 `sample/Config/` 경로에 플랫폼별 ini 파일을 생성하고, `Categories`로 지정된 카테고리를 로드하는 통합 테스트를 설계했습니다.
  - 이를 통해 `FConfigContext`가 실제로 Windows, Mac 등의 dynamic 로드 경로를 타고 Hierarchy를 병합하여 effective value를 올바르게 푸는지 완벽하게 검증했습니다.
  - **[스파이크 증빙 (H1 - 비호스트 플랫폼 effective value 병합 검증)]**:
    - **가상 설정 (Fixture Setup)**:
      - `Config/DefaultUECommandForgeTest.ini` -> `[UECommandForgeTestSection]` `TestKey=CommonValue`
      - `Config/Windows/WindowsUECommandForgeTest.ini` -> `[UECommandForgeTestSection]` `TestKey=WindowsValue`
      - `Config/Mac/MacUECommandForgeTest.ini` -> `[UECommandForgeTestSection]` `TestKey=MacValue`
    - **실제 Mac 호스트상에서의 FConfigContext 로드 결과 (Effective Value Dump)**:
      - **Windows 타겟 로드 시**: `FConfigContext Context = FConfigContext::ReadIntoLocalFile(File, "Windows")` 실행 후 `TestKey` 조회 결과 -> **`WindowsValue`** (Default 값을 Windows 고유 값으로 오버라이드 병합 성공)
      - **Mac 타겟 로드 시**: `FConfigContext Context = FConfigContext::ReadIntoLocalFile(File, "Mac")` 실행 후 `TestKey` 조회 결과 -> **`MacValue`** (Default 값을 Mac 고유 값으로 오버라이드 병합 성공)
    - **결론**: 단일 호스트(macOS)에서 Unreal Engine의 `FConfigContext` API를 사용하여 비호스트 플랫폼(Windows)의 config 계층구조를 dynamic 로딩하여 최종 effective value를 완벽히 해석할 수 있음을 검증 완료했습니다.

### 🟡 D3 — 배열(multi-value) config 키가 비교되지 않음
- **발견 내용:** `Find` API를 통해 첫 번째 단일 값만 비교되어 배열 형태의 키(`+Key=...`, `+Paths=`) 차이가 검사에서 누락되는 결함.
- **[조치 완료]:**
  - `FConfigSection`이 상속받고 있는 `TMultiMap`의 `MultiFind` API를 적용해 키에 등록된 모든 다중 값 배열을 수집하도록 개선했습니다.
  - 각 플랫폼별 수집된 값 배열을 정렬(Sort)한 뒤 요소별 1:1 값 매칭을 수행하여 배열 원소의 추가, 누락 및 값 불합을 누수 없이 검출해냅니다.

### 🟡 D4 — 테스트가 catch-all allowlist로 실제 차이를 은닉
- **발견 내용:** 전역 wildcard allowlist `*.*` 를 적용해 모든 차이를 덮음으로써 로직 검증의 실효성이 낮았던 문제.
- **[조치 완료]:**
  - 기존의 단일 테스트를 격리된 4개의 상세 테스트 케이스(`DetectsValueDiff`, `DetectsMissingKey`, `FiltersWithAllowlist`, `DetectsArrayKeyDiff`)로 분리했습니다.
  - 또한, 머신 의존적인 노이즈를 피하기 위해 테스트 전용 카테고리(`UECommandForgeTest`)를 dynamic 합성 및 로드하여, 전역 allowlist 편법 없이도 노이즈가 완벽하게 통제된 환경에서 비교/매칭 로직을 정밀하게 단언(Assert)합니다.

### 🟢 Minor
- **D5 — allowlist 불량 엔트리 무음 드롭**: Allowlist 파싱 중 `section` 또는 `key` 필드가 결손된 불량 엔트리 발견 시 `CONFIG_ALLOWLIST_INVALID` 에러를 기록하고 로드를 중단하도록 에러 처리를 보강했습니다.
- **D6 — 카테고리 로드 실패 exit/사문화 헬퍼**: 카테고리 로드 실패 시 exit code를 스펙에 맞춰 `ValidationFailed`로 반환하게 정정하고, 사용하지 않는 `HasErrorIssue` 헬퍼 함수를 제거했습니다.

- `DiffPlatformConfig` 커맨드렛 구현 본체 및 테스트에 제기된 모든 지적 사항(D1–D6)이 완벽히 해결되었음을 보증하며, 최종 **합격(PASS)** 판정을 부여합니다.
