# C++ 클래스 배치 생성 3차 재리뷰 보고서 (2026-06-09_reviews_03)

본 문서는 `GenerateCppClassBatch` 변경사항에 대한 3차 재리뷰 결과를 기록한다. 2차 재리뷰 이후 preflight 정규화 중복 검사, 롤백 실패 감지, rollback 테스트 seam, trailing whitespace 정리가 반영된 상태를 기준으로 검토했다.

## 2. 이전 재리뷰 대비 상태

- preflight 중복 경로 계산 불일치:
  - `GenerateSingleClass(..., false, ...)` 결과의 `header_path`, `source_path`를 사용하도록 바뀌어 경로 정규화 문제는 해소됐다.
- rollback 테스트가 preflight만 검증하던 문제:
  - 디렉터리 blocker fixture로 실제 apply 중 실패 경로를 타도록 보강됐다.
- 신규 파일 trailing whitespace:
  - 신규 C++/wrapper 파일 기준으로 해소됐다.
- 롤백 실패 감지:
  - 이전 성공 파일 및 `Build.cs` 롤백 실패 감지는 추가됐다.
  - 다만 실패 클래스 자체의 부분 생성물 cleanup 실패는 아직 감지하지 못한다.

---

## 3. 주요 발견 사항

### HIGH: 파일 경로 중복만 검사하고 Unreal C++ 심볼 중복은 놓침

- 위치: `GenerateCppClassBatchCommandlet.cpp:132`, `GenerateCppClassCommandlet.cpp:771`
- 내용:
  배치 preflight는 `header_path`, `source_path` 기준으로만 중복을 검사한다. 하지만 실제 C++ 클래스 심볼은 `EnsureUnrealClassSymbol` 로직을 통해 `Foo` Actor가 `AFoo`로 변환된다.
- 위험:
  배치에 `Foo`와 `AFoo`가 함께 있으면 파일 경로는 `Foo.h`와 `AFoo.h`로 달라 preflight를 통과한다. 그러나 생성되는 UCLASS 심볼은 둘 다 `AFoo`가 되어 UHT 또는 컴파일 단계에서 중복 클래스 오류가 발생한다. Commandlet은 파일 생성 단계에서 `ok=true`로 보고할 수 있으므로, 이후 빌드에서 지연 실패가 발생한다.
- 권장 조치:
  preflight에서 `class_symbol` validation 값을 기준으로도 batch 내부 중복을 검사한다. `header_path/source_path` 중복 검사와 별도로 `class_symbol` `TSet`을 유지하고, 중복 시 `DUPLICATE_CLASS_SYMBOL` 같은 error issue를 추가한다.

### HIGH: 실패한 클래스 자체의 partial file cleanup 실패를 배치 rollback이 감지하지 못함

- 위치: `GenerateCppClassCommandlet.cpp:875`, `GenerateCppClassBatchCommandlet.cpp:261`
- 내용:
  `GenerateSingleClass`는 header/source 중 하나라도 저장 실패하면 두 파일을 삭제하지만, `IFileManager::Delete` 반환값을 확인하지 않는다. 배치 commandlet은 이전 성공 클래스의 `ChangedFiles`만 `CreatedFiles`에 추적하므로 실패한 클래스에서 일부 파일이 남아도 batch rollback 검증 대상에 포함되지 않는다.
- 위험:
  예를 들어 실패 클래스의 header는 저장됐고 source 저장이 실패한 뒤 header 삭제도 실패하면, 실제 워킹트리에 실패 클래스의 partial artifact가 남을 수 있다. 하지만 최종 리포트는 `partial_changes_remaining=false`로 잘못 보고할 수 있다.
- 권장 조치:
  `GenerateSingleClass`의 write failure cleanup 결과를 검증하고 실패한 cleanup 경로를 `OutReport.ValidationIssues` 또는 `OutReport.Validation`에 기록한다. 또는 배치 commandlet이 실패한 `ClassReport`의 `header_path`, `source_path`를 이용해 실패 클래스의 residual file 존재 여부까지 확인해야 한다.

### MEDIUM: 롤백 실패 시 표준 Result 필드만 보는 소비자가 오해할 수 있음

- 위치: `GenerateCppClassBatchCommandlet.cpp:294`, `GenerateCppClassBatchCommandlet.cpp:317`
- 내용:
  롤백 실패 시 `rollback_failed=true`, `partial_changes_remaining=true`를 validation에 기록하지만, 표준 필드인 `applied`는 `false`로 유지하고 `changed_files`는 비운다.
- 위험:
  Result JSON 소비자가 `applied`와 `changed_files`만 보면 실제 변경이 남아 있는 상황을 “적용 없음”으로 오해할 수 있다.
- 권장 조치:
  롤백 실패로 변경이 남은 경우 `changed_files`에 잔존 파일을 기록하거나, `applied` 의미를 프로젝트 표준에 맞춰 재정의해야 한다. 최소한 실패 리포트에서 남은 변경 파일 경로를 machine-readable한 표준 배열에 남기는 편이 안전하다.

### LOW: 롤백 리포트 계약 검증이 더 필요함

- 위치: `GenerateCppClassBatchCommandletTest.cpp:315`
- 내용:
  rollback test는 apply 중 실패와 이전 성공 파일 삭제를 검증하지만, 리포트의 `rollback_failed=false`, `partial_changes_remaining=false` 및 실패 클래스 partial artifact 부재까지는 검증하지 않는다.
- 권장 조치:
  새로 추가한 rollback validation 계약을 테스트로 고정한다. 특히 실패 클래스의 header/source 잔존 여부와 `partial_changes_remaining` 값을 함께 확인한다.

## 4. 검증 및 한계

- 실행한 검증:
  - `git diff --check`
  - 신규 파일 대상 trailing whitespace 검색
  - 관련 `.sh` 파일 `bash -n`
  - C++ 리뷰 서브에이전트 기반 교차 검토
- 확인 결과:
  - `git diff --check` 통과
  - 신규 C++/wrapper 파일 trailing whitespace 없음
  - 관련 shell script syntax check 통과
- 실행하지 않은 검증:
  - Unreal build
  - Unreal Automation Test
  - smoke script 실제 실행
  - `clang-tidy`, `cppcheck`
- 한계:
  - 이번 요청은 리뷰 문서 작성이므로 코드 수정과 테스트 실행은 수행하지 않았다.
  - 바이너리 `.uasset` 및 `.umap` 변경은 내부 의미를 해석하지 않았다.

---

## 5. 후속 권장 순서

1. batch preflight에 `class_symbol` 중복 검사를 추가한다.
2. 실패 클래스의 partial artifact cleanup 실패를 `GenerateSingleClass` 또는 batch rollback에서 감지한다.
3. 롤백 실패 시 표준 Result JSON 필드에 잔존 변경 파일을 명확히 남긴다.
4. rollback validation 계약과 실패 클래스 residual file 부재를 테스트로 검증한다.
5. Unreal Automation Test와 smoke script를 실행해 실제 commandlet 동작을 확인한다.
