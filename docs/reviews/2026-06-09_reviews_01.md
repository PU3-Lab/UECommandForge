## 2. 주요 발견 사항

### HIGH: 배치 apply 실패 시 부분 생성물이 남을 수 있음

- 위치: `GenerateCppClassBatchCommandlet.cpp:105`, `GenerateCppClassCommandlet.cpp:849`
- 내용:
  `GenerateCppClassBatchCommandlet`은 `BatchSpec.Classes`를 순회하면서 각 항목마다 `GenerateSingleClass(..., bApply, ...)`를 바로 호출한다. `GenerateSingleClass`는 `bApply=true`일 때 `.h`, `.cpp`, 필요 시 `Build.cs` 변경을 즉시 디스크에 쓴다.
- 위험:
  1번 클래스가 성공적으로 파일을 쓴 뒤 2번 클래스가 invalid path, existing file, missing Build.cs 등으로 실패하면 최종 리포트는 `ok=false`가 되지만 앞선 생성 파일 또는 `Build.cs` 변경은 워킹트리에 남는다. 즉 배치 실패가 원자적으로 실패하지 않는다.
- 추가 사례:
  같은 배치 안에 동일 class/output 조합이 중복된 경우 dry-run에서는 둘 다 파일 미존재 상태로 `ok=true`가 될 수 있다. apply에서는 첫 번째 항목이 파일을 생성한 뒤 두 번째 항목이 `CPP_CLASS_FILE_EXISTS`로 실패하여 부분 적용 상태가 된다.
- 권장 조치:
  배치 실행 전에 전체 preflight 검증을 먼저 수행하고, batch 내부 target 중복 검사를 추가한다. apply 단계에서 실패가 발생하면 이미 쓴 `.h`, `.cpp`, `Build.cs` 변경을 롤백하는 구조가 필요하다.

### MEDIUM: 실패 및 롤백 회귀 테스트가 부족함

- 위치: `GenerateCppClassBatchCommandletTest.cpp:12`
- 내용:
  신규 batch 테스트는 dry-run 성공과 apply 성공만 검증한다. 부분 적용 회귀를 잡을 실패 케이스가 없다.
- 필요한 테스트:
  - 첫 번째 클래스 생성 성공 후 두 번째 클래스 실패 시 앞선 `.h/.cpp`가 남지 않는지 검증
  - 같은 batch 안의 동일 class/output 중복을 dry-run 또는 preflight에서 잡는지 검증
  - `-ApplyBuildCs=true`로 `Build.cs`가 변경된 뒤 후속 항목이 실패하면 `Build.cs`가 원복되는지 검증
  - dry-run과 apply의 검증 결과가 동일한 실패 조건을 보고하는지 검증

### LOW: EOF extra blank line 경고

- 위치: `GenerateCppClassCommandlet.cpp:988`, `GenerateCppClassCommandlet.h:31`
- 내용:
  `git diff --check`에서 EOF의 extra blank line 경고가 발생했다.
- 권장 조치:
  기능 문제는 아니지만 커밋 전 불필요한 EOF 빈 줄을 제거한다.

## 3. 보조 관찰 사항

- `GenerateSingleClass` 분리는 단건 커맨드렛과 배치 커맨드렛의 생성 로직 중복을 줄이는 방향으로 타당하다.
- 현재 배치 파서는 `includes`, `properties`, `functions`, `build_dependencies`를 공통 설정으로 파싱하여 모든 class에 적용한다. 의도된 스펙이라면 문서화가 필요하고, class별 속성/함수가 필요하면 스펙 구조 확장이 필요하다.
- `tools/ue/generate_cpp_class_batch.sh`는 기존 `run_commandlet.sh`에 `GenerateCppClassBatch`와 `-Spec`을 전달하는 얇은 wrapper로 구성되어 있다.
- `tools/test/smoke/cpp_generation_batch.sh`는 `jq`와 `ls -t | head -1`에 의존한다. 로컬 smoke 환경에서는 괜찮지만, CI 환경에서는 `jq` 설치 여부를 확인하거나 의존성을 명시하는 편이 좋다.

## 4. 검증 및 한계

- 실행한 검증:
  - `git diff --check`
- 확인 결과:
  - `GenerateCppClassCommandlet.cpp`, `GenerateCppClassCommandlet.h`에서 EOF extra blank line 경고 확인
- 실행하지 않은 검증:
  - Unreal Automation Test
  - smoke script
  - `clang-tidy`, `cppcheck`
- 한계:
  - 사용자가 리뷰 문서화를 요청한 범위이므로 코드 수정과 테스트 실행은 수행하지 않았다.
  - `.uasset` 및 `.umap` 변경은 바이너리 파일이라 내부 의미를 해석하지 않았다.

---

## 5. 후속 권장 순서

1. `GenerateCppClassBatchCommandlet`에 전체 preflight 검증 단계를 추가한다.
2. batch 내부에서 생성 대상 header/source path 중복을 검사한다.
3. apply 중 실패 시 이미 쓴 파일과 `Build.cs` 변경을 롤백한다.
4. 실패 및 롤백 케이스 자동화 테스트를 추가한다.
5. EOF extra blank line 경고를 정리한 뒤 `git diff --check`와 관련 Unreal Automation Test를 재실행한다.
