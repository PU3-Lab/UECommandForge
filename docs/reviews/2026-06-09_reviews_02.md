## 2. 1차 리뷰 대비 상태

- EOF extra blank line 경고는 해소됐다. `git diff --check`는 통과했다.
- 배치 partial apply 문제는 preflight와 롤백 구조 추가로 개선 시도가 있었다.
- 다만 롤백 실패 감지, preflight와 실제 생성 경로의 정규화 불일치, 테스트가 실제 rollback block을 타지 않는 문제가 새로 확인되어 현재 변경은 승인하기 어렵다.

---

## 3. 주요 발견 사항

### HIGH: 롤백 실패를 감지하지 않고 적용 없음으로 보고함

- 위치: `GenerateCppClassBatchCommandlet.cpp:264`
- 내용:
  apply 실패 후 생성 파일 삭제(`Delete`)와 `Build.cs` 복구(`SaveStringToFile`)를 수행하지만 반환값을 확인하지 않는다. 이후 `Report.bApplied=false`, `Report.ChangedFiles.Empty()`로 리포트한다.
- 위험:
  파일 잠금, 권한 문제, 디스크 오류 등으로 롤백이 실패하면 실제 워킹트리에는 일부 `.h/.cpp` 또는 `Build.cs` 변경이 남을 수 있다. 하지만 Result JSON은 적용이 없었던 것처럼 보이므로 후속 자동화가 오판할 수 있다.
- 권장 조치:
  롤백 작업의 성공 여부를 모두 검사하고, 실패한 파일 경로를 `errors` 또는 `issues`에 기록한다. 롤백 실패 시에는 `bApplied=false`로 단정하지 말고, `rollback_failed=true`, `partial_changes_remaining=true` 같은 명확한 validation 값을 남긴다.

### HIGH: preflight 중복 경로 계산이 실제 생성 경로 정규화와 다름

- 위치: `GenerateCppClassBatchCommandlet.cpp:123`, `GenerateCppClassCommandlet.cpp:735`
- 내용:
  배치 preflight의 중복 검사 경로는 raw `Name`, `Module`, `OutputPath`를 사용해 계산한다. 실제 생성 헬퍼는 `Name`, `Type`, `Module`을 trim하고 `OutputPath`를 `NormalizeRelativePath`로 정규화한 뒤 파일 경로를 만든다.
- 위험:
  예를 들어 `"Weapon"`과 `" Weapon "`은 preflight 중복 검사에서 다른 target으로 보일 수 있지만, 실제 생성에서는 같은 class/file target이 된다. dry-run에서 성공한 스펙이 apply에서 실패/롤백 경로로 들어갈 수 있어 dry-run 신뢰성이 깨진다.
- 권장 조치:
  중복 검사도 `GenerateSingleClass`와 동일한 정규화 함수를 사용해야 한다. 가장 안전한 방식은 dry-run `GenerateSingleClass(..., false, ...)` 결과의 `header_path`, `source_path` validation 값을 기반으로 중복을 검사하는 것이다.

### HIGH: 롤백 테스트가 실제 apply 중 실패와 rollback block을 검증하지 않음

- 위치: `GenerateCppClassBatchCommandletTest.cpp:305`, `GenerateCppClassBatchCommandletTest.cpp:411`
- 내용:
  `RollbackOnError` 테스트는 두 번째 class name을 `123InvalidName`으로 설정한다. 이 오류는 preflight 단계에서 이미 잡히므로 apply 루프와 rollback block은 실행되지 않는다. `RollbackBuildCsOnError` 테스트도 같은 invalid class name 때문에 `Build.cs`가 실제 변경되기 전에 실패할 수 있다.
- 위험:
  테스트 이름은 롤백을 검증하는 것처럼 보이지만 실제로는 preflight 실패 검증에 가깝다. 따라서 apply 도중 파일 쓰기 또는 `Build.cs` 변경 후 실패했을 때 롤백이 작동하는지 보장하지 못한다.
- 권장 조치:
  preflight는 통과하지만 apply 단계에서 실패하는 조건을 별도로 만들어야 한다. 예를 들어 두 번째 파일 쓰기 실패를 유도할 수 있는 권한/경로 fixture, `GenerateSingleClass`의 쓰기 실패 경로를 직접 검증할 수 있는 seam, 또는 테스트 전용 commandlet 옵션을 사용해 apply 중간 실패를 강제로 발생시키는 방식이 필요하다.

### LOW: 신규 untracked 파일의 trailing whitespace

- 위치 예시:
  - `CppClassBatchSpec.h:15`
  - `GenerateCppClassBatchCommandlet.cpp:125`
  - `GenerateCppClassBatchCommandletTest.cpp:115`
- 내용:
  신규 untracked 파일들에 trailing whitespace가 남아 있다. `git diff --check`는 현재 tracked diff 위주로 보이기 때문에 신규 파일 공백은 별도 검사로 확인했다.
- 권장 조치:
  커밋 전 신규 파일 전체의 trailing whitespace를 제거한다.

## 4. 보조 관찰 사항

- `GenerateSingleClass` 분리는 유지보수 측면에서 타당하다.
- `GenerateCppClassBatchCommandlet`이 preflight와 apply를 분리한 방향은 1차 리뷰의 핵심 우려를 줄이는 방향이다.
- 현재 배치 spec은 `includes`, `properties`, `functions`, `build_dependencies`를 공통 설정으로만 제공한다. class별 설정이 필요하면 별도 스펙 확장이 필요하다.
- shell wrapper는 기존 단건 wrapper와 유사한 얇은 계층이며, 별도 기능 리스크는 크지 않아 보인다.

## 5. 검증 및 한계

- 실행한 검증:
  - `git diff --check`
  - 신규 파일 대상 trailing whitespace 검색
  - C++ 리뷰 서브에이전트 기반 교차 검토
- 확인 결과:
  - `git diff --check` 통과
  - 신규 untracked 파일에서 trailing whitespace 다수 확인
- 실행하지 않은 검증:
  - Unreal Automation Test
  - smoke script
  - `clang-tidy`, `cppcheck`
- 한계:
  - 이번 요청은 리뷰 문서 작성이므로 코드 수정과 테스트 실행은 수행하지 않았다.
  - 바이너리 `.uasset` 및 `.umap` 변경은 내부 의미를 해석하지 않았다.

## 6. 후속 권장 순서

1. preflight 중복 검사에 실제 생성과 동일한 정규화 경로를 사용한다.
2. 롤백 작업의 실패 여부를 감지하고 Result JSON에 명확히 기록한다.
3. preflight 실패 테스트와 apply 중간 실패/롤백 테스트를 분리한다.
4. `Build.cs`가 실제 변경된 뒤 실패하는 테스트 fixture를 만든다.
5. 신규 파일 trailing whitespace를 정리한다.
6. 관련 Unreal Automation Test와 smoke script를 실행해 회귀를 확인한다.
