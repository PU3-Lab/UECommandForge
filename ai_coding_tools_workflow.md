# AI 코딩 도구 역할 분담 및 개발 워크플로

## 1. 목적

본 문서는 Codex, Antigravity, Claude Code를 함께 사용할 때 각 도구의 역할을 분리하고, 중복 작업과 토큰 낭비를 줄이기 위한 개발 워크플로를 정의한다.

기본 전략은 다음과 같다.

> **설계: Codex → 구현: Antigravity → 리뷰: Claude Code**

구현자와 리뷰어를 분리하고, 각 도구가 잘하는 작업에 집중하도록 구성한다.

---

## 2. 도구별 역할

| 도구 | 기본 역할 | 주요 작업 |
|---|---|---|
| **Codex** | 설계자·작업 관리자 | 요구사항 분석, 영향 범위 조사, 구현 계획, 작업 분할, 수락 조건, 테스트 계획 |
| **Antigravity** | 메인 구현자 | 코드 작성, 로컬 실행, UI 수정, 브라우저 검증, 반복 수정 |
| **Claude Code** | 독립 리뷰어·난제 해결자 | 코드 리뷰, 구조 검증, 버그 분석, 복잡한 리팩터링, 예외 처리 점검 |

### 권장 사용 비중

```text
Codex        30%  — 설계, 조사, 작업 분할, 계획 검증
Antigravity  45%  — 일반 구현, UI, 실행 검증
Claude Code  25%  — 복잡한 구현, 디버깅, 코드 리뷰
```

---

## 3. 기본 개발 흐름

```text
요구사항 또는 이슈 정의
        ↓
Codex: 요구사항 분석 및 구현 계획 작성
        ↓
Antigravity: 계획 기준 구현 및 실행 검증
        ↓
Claude Code: git diff 또는 PR 독립 리뷰
        ↓
Antigravity: 리뷰 반영 및 테스트 재실행
        ↓
Codex: 최초 설계와 구현 결과 일치 여부 확인
        ↓
PR Merge
```

---

## 4. Codex 역할

Codex는 단순한 설계 문서 작성 도구가 아니라, 구현 전에 작업의 경계를 명확히 만드는 역할을 맡는다.

### Codex에 맡길 작업

- 요구사항 분석
- 기존 코드 영향 범위 조사
- 변경 대상 파일 및 모듈 식별
- 구현 순서 정의
- 작업 단위 분할
- 수락 조건 작성
- 테스트 시나리오 작성
- 위험 요소 및 회귀 가능성 정리
- 구현 완료 후 계획 준수 여부 확인

### Codex 설계 문서 템플릿

```markdown
# Feature: 기능명

## 목적
왜 필요한가?

## 범위
포함되는 작업과 포함되지 않는 작업

## 변경 대상
- 파일
- 모듈
- API
- DB
- UI

## 구현 계획
1. 작업 1
2. 작업 2
3. 작업 3

## 수락 조건
- [ ] 정상 동작 조건
- [ ] 실패 처리 조건
- [ ] 회귀 방지 조건

## 테스트 계획
- 정상 케이스
- 실패 케이스
- 경계값
- 회귀 테스트

## 위험 요소
- 구조 변경 영향
- 성능 영향
- 데이터 호환성
- 롤백 필요 여부
```

### Codex 사용 시 주의점

- 설계 문서는 구현자가 그대로 사용할 수 있을 정도로 구체적으로 작성한다.
- 작업 단위는 PR 하나에 담을 수 있는 크기로 나눈다.
- 서로 의존성이 높은 작업을 무리하게 병렬화하지 않는다.
- 설계만 하고 끝내지 말고, 구현 완료 후 최초 계획과 실제 결과를 비교한다.

---

## 5. Antigravity 역할

Antigravity는 메인 구현 도구로 사용한다. 코드 작성뿐 아니라 실행 결과를 확인하고, UI와 브라우저 흐름을 반복 검증하는 데 집중한다.

### Antigravity에 맡길 작업

- 일반 기능 구현
- 프런트엔드 UI 수정
- 브라우저 동작 확인
- API 연동 확인
- 로컬 로그 기반 수정
- 작은 단위 리팩터링
- 리뷰 지적 사항 반영
- 테스트 재실행

### Antigravity가 특히 적합한 작업

- Next.js, Flutter 등 UI 중심 작업
- 웹 브라우저에서 바로 확인 가능한 기능
- API 호출 결과를 화면에서 검증하는 작업
- 작은 수정과 실행 확인이 반복되는 작업
- Firebase, Google Cloud 연동 작업

### Antigravity 사용 시 주의점

- Codex 설계 문서를 기준으로 구현한다.
- 구현 중 설계 변경이 필요하면 임의로 크게 바꾸지 말고 변경 이유를 기록한다.
- 한 번에 너무 많은 파일을 수정하지 않는다.
- 리뷰 전 최소한의 테스트와 실행 검증을 완료한다.

---

## 6. Claude Code 역할

Claude Code는 구현에 직접 참여하지 않은 독립 리뷰어로 사용한다. 구현자가 놓친 구조적 문제, 예외 처리, 테스트 누락을 찾는 역할을 맡는다.

### Claude Code에 맡길 작업

- PR 또는 git diff 리뷰
- 설계 의도와 구현 결과 비교
- 예외 처리 누락 확인
- 테스트 부족 확인
- 과도한 복잡성 확인
- 보안 및 안정성 점검
- 대규모 리팩터링 검토
- 원인을 찾기 어려운 버그 분석

### Claude Code 리뷰 체크리스트

```text
- 요구사항을 정확히 충족하는가?
- 설계 문서와 구현 결과가 일치하는가?
- 불필요한 복잡성이 추가되지 않았는가?
- 예외 처리와 실패 복구가 충분한가?
- 테스트가 정상·실패·경계값을 포함하는가?
- 기존 기능에 회귀 위험이 없는가?
- 성능, 보안, 유지보수성 문제가 없는가?
```

### Claude Code 사용 시 주의점

- 리뷰 단계에서는 직접 수정부터 시키지 말고, 먼저 문제점과 우선순위를 정리하게 한다.
- 리뷰 결과는 치명적 문제, 중요 문제, 개선 제안으로 구분한다.
- 복잡한 구현이나 반복 실패 작업은 예외적으로 Claude Code에 구현을 넘길 수 있다.

---

## 7. 역할 전환 기준

기본 흐름은 설계 → 구현 → 리뷰를 유지하되, 작업 난이도에 따라 역할을 전환한다.

### Antigravity에서 Claude Code 구현으로 넘길 기준

- 동일한 오류를 두 번 이상 반복해서 해결하지 못함
- 여러 모듈이 연결된 복잡한 백엔드 흐름
- 대규모 리팩터링
- 트랜잭션, 큐, 재시도, 동시성 처리
- 원인 파악이 어려운 버그
- UE5 C++ 플러그인처럼 컴파일과 구조 이해가 중요한 작업

### Codex에 추가로 맡길 작업

- 독립 기능의 병렬 조사
- 테스트 케이스 보강
- 문서화
- 구현 완료 후 계획 검증
- PR 범위가 너무 커졌는지 확인

## 9. PR 운영 규칙

### 좋은 PR 단위

```text
- Record API 추가
- Record API 테스트 추가
- Record 확인 UI 추가
```

### 피해야 할 PR 단위

```text
- Record API
- UI
- DB 마이그레이션
- Queue 구조 변경
- 대규모 리팩터링
```

### 권장 규칙

- PR 하나에는 하나의 의미 있는 변경만 포함한다.
- 설계 문서의 수락 조건을 PR 설명에 포함한다.
- 구현자와 리뷰어를 분리한다.
- 리뷰 전에 자동 테스트와 기본 실행 검증을 완료한다.
- 리뷰 지적 사항은 같은 구현 도구에서 반영한다.

---

## 10. 최종 운영 원칙

1. **같은 작업을 세 도구에 동시에 구현시키지 않는다.**
2. **Codex는 작업의 경계와 완료 조건을 명확히 한다.**
3. **Antigravity는 실제 구현과 실행 검증을 담당한다.**
4. **Claude Code는 독립적인 시각으로 리뷰한다.**
5. **복잡한 작업만 Claude Code 구현으로 전환한다.**
6. **PR은 작게 유지한다.**
7. **설계 문서, 구현 결과, 리뷰 결과를 동일한 기준으로 관리한다.**

---

## 11. 요약

기본 운영 모델은 다음과 같다.

> **Codex가 설계하고, Antigravity가 구현하며, Claude Code가 리뷰한다.**

이 조합은 각 도구의 역할을 분리해 중복 작업을 줄이고, 구현자와 리뷰어를 분리해 코드 품질을 높이는 데 목적이 있다. 다만 복잡한 백엔드 흐름, 대규모 리팩터링, UE5 C++ 작업처럼 난도가 높은 구현은 Claude Code로 전환한다.
